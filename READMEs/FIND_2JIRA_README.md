# `findings_to_jira.py`

Publish Black Duck vulnerability rollup data into Jira.

The script supports two publishing modes:

1. **Flat findings mode**: create one Jira issue per unique vulnerability rollup finding from `findings.csv`.
2. **Hierarchy plan mode**: create Jira Epic/Story/Vulnerability issues from a generated hierarchy plan JSON.

The script builds Jira issue payloads, checks Jira for existing issues by deterministic lookup labels, writes result/plan files, and maintains a local state cache to prevent duplicates across reruns.

## Inputs

| File | Purpose |
|---|---|
| `findings.csv` | Rollup findings generated by the Black Duck vulnerability rollup workflow. Used in flat findings mode. |
| `jira-rollup-config.json` | Jira project, issue type, labels, priority mapping, dedupe, AI, and hierarchy publishing settings. |
| `jira-rollup-state.json` | Local cache of created/found Jira issues and hierarchy links. |
| `jira-hierarchy-plan.json` | Optional hierarchy plan generated by hierarchy planning tooling. Used when `--hierarchy-plan` is supplied. |

## Auth

Use environment variables for Jira credentials.

### Basic auth

```bash
export JIRA_USER="user@example.com"
export JIRA_API_TOKEN="..."
```

### Bearer auth

```bash
export JIRA_PAT="..."
```

Set `"jira.auth_mode": "bearer"` in the config file when using `JIRA_PAT`.

`JIRA_URL` can override `jira.url` from the config file.

```bash
export JIRA_URL="https://your-domain.atlassian.net"
```

If Jira URL/auth is incomplete, the script automatically runs in dry-run mode.

## Config overview

Minimal flat-publishing config:

```json
{
  "jira": {
    "url": "https://your-domain.atlassian.net",
    "project_key": "SEC",
    "issue_type": "Task",
    "api_version": "2",
    "auth_mode": "basic",
    "verify_tls": true
  },
  "issue": {
    "summary_template": "[Black Duck] {severity} {vulnerability} in {component} - {parent_project} {parent_version}",
    "labels": ["blackduck", "subproject_rollup"],
    "priority_by_severity": {
      "CRITICAL": "Highest",
      "HIGH": "High",
      "MEDIUM": "Medium",
      "LOW": "Low"
    },
    "additional_fields": {}
  },
  "dedupe": {
    "label_prefix": "bd_rollup_",
    "hash_length": 24
  }
}
```

Hierarchy-specific config options:

```json
{
  "hierarchy": {
    "epic_issue_type": "Epic",
    "story_issue_type": "Story",
    "vulnerability_issue_type": "Task",
    "story_parent_mode": "jira_parent",
    "vulnerability_parent_mode": "issue_link",
    "issue_link_type": "Relates",
    "epic_link_field": "",
    "labels": {
      "epic": ["bd_rollup_parent"],
      "story": ["bd_rollup_child"],
      "vulnerability": ["bd_rollup_vuln"]
    },
    "additional_fields": {
      "epic": {},
      "story": {},
      "vulnerability": {}
    }
  }
}
```

### Parent modes

| Mode | Description |
|---|---|
| `jira_parent` | Adds Jira `parent.key` to the issue create payload. |
| `issue_link` | Creates the issue first, then links it to its parent using `hierarchy.issue_link_type`. |
| `epic_link_field` | Sets a configured custom Epic Link field. Requires `hierarchy.epic_link_field`. |

Defaults:

- Stories use `jira_parent`.
- Vulnerability issues use `issue_link`.

## Optional AI descriptions

AI-generated descriptions are disabled by default. Enable them in config:

```json
{
  "ai": {
    "enabled": true,
    "base_url": "https://api.openai.com/v1",
    "model": "gpt-4o-mini",
    "api_key_env": "OPENAI_API_KEY"
  }
}
```

Then set the configured environment variable:

```bash
export OPENAI_API_KEY="..."
```

If AI is enabled but the API key is missing, or generation fails, the script falls back to the built-in Jira wiki description template.

## Flat findings dry run

Dry run is the default unless `--apply` is provided.

```bash
python findings_to_jira.py \
  --findings findings.csv \
  --config jira-rollup-config.json \
  --state jira-rollup-state.json \
  --results-out jira-rollup-results.csv \
  --plan-out jira-rollup-plan.json
```

## Flat findings apply run

Creates Jira issues for findings that are not already in local state or Jira.

```bash
python findings_to_jira.py \
  --findings findings.csv \
  --config jira-rollup-config.json \
  --state jira-rollup-state.json \
  --results-out jira-rollup-results.csv \
  --plan-out jira-rollup-plan.json \
  --apply
```

## Hierarchy plan dry run

When `--hierarchy-plan` is supplied, the script publishes hierarchy nodes instead of flat findings.

```bash
python findings_to_jira.py \
  --hierarchy-plan jira-hierarchy-plan.json \
  --config jira-rollup-config.json \
  --state jira-rollup-state.json \
  --results-out jira-hierarchy-publish-test-results.csv \
  --plan-out jira-hierarchy-publish-test-plan.json
```

## Hierarchy plan apply run

Creates Epic/Story/Vulnerability issues and any configured issue links.

```bash
python findings_to_jira.py \
  --hierarchy-plan jira-hierarchy-plan.json \
  --config jira-rollup-config.json \
  --state jira-rollup-state.json \
  --results-out jira-hierarchy-publish-test-results.csv \
  --plan-out jira-hierarchy-publish-test-plan.json \
  --apply
```

Hierarchy nodes are processed in this order:

1. Epic
2. Story
3. Vulnerability

This allows child nodes to reference parent issue keys when using `jira_parent`, `epic_link_field`, or `issue_link`.

## Duplicate-stop test by rollup hash

Use the full SHA-256 rollup hash when testing a known Jira duplicate.

```bash
python findings_to_jira.py \
  --findings findings.csv \
  --config jira-rollup-config.json \
  --state jira-rollup-state.json \
  --results-out jira-rollup-results-duplicate-test.csv \
  --plan-out jira-rollup-plan-duplicate-test.json \
  --apply \
  --refresh-existing \
  --only-rollup-key-hash "794762591e2730b5212f6f256c8c38accacbb235c13e577314a56bf7cbf48771" \
  --max-create 1 \
  --debug
```

Expected duplicate-stop summary:

```text
Unique rollup findings:  1
Created:                 0
Skipped existing:        1
Errors:                  0
```

## Targeting filters

| Flag | Description |
|---|---|
| `--only-parent-project` | Process only findings or hierarchy nodes for an exact parent project name. |
| `--only-parent-version` | Process only findings or hierarchy nodes for an exact parent version. |
| `--only-subproject` | Process only findings or hierarchy nodes for an exact subproject name. |
| `--only-vulnerability` | Process only findings or hierarchy nodes for an exact vulnerability ID. |
| `--only-rollup-key` | Flat findings mode only: process one exact raw pipe-delimited `rollup_key`. |
| `--only-rollup-key-hash` | Flat findings mode only: process one finding by full SHA-256 hash of `rollup_key`. Preferred for copied Jira Rollup IDs. |
| `--limit` | Process only the first N findings or hierarchy nodes after filters. |

In hierarchy mode, filtering preserves required ancestor nodes so selected child nodes can still be attached to their parents.

## Jira creation controls

| Flag | Description |
|---|---|
| `--apply` | Actually create Jira issues. Without it, the script is dry-run only. |
| `--dry-run` | Force dry-run even if `--apply` is present. |
| `--refresh-existing` | Query Jira even if the issue exists in local state. Useful for duplicate verification. |
| `--max-create` | Safety cap for created Jira issues in one run. |
| `--description-format` | `wiki` or `adf`. Use `adf` when required by Jira Cloud/API behavior. |
| `--jql-label-batch-size` | Number of lookup labels per Jira search batch. |

## Output files

| Flag | Default | Description |
|---|---:|---|
| `--results-out` | `jira-rollup-results.csv` | CSV report of each action taken. |
| `--plan-out` | `jira-rollup-plan.json` | JSON dry-run/create plan output for audit/debugging. |
| `--state` | `jira-rollup-state.json` | Persistent local dedupe cache keyed by external ID. |

Common result actions include:

| Action | Meaning |
|---|---|
| `would_create` | Dry-run planned issue creation. |
| `created` | Jira issue was created. |
| `skip_existing_state` | Existing issue was found in local state. |
| `skip_existing_jira` | Existing issue was found in Jira by lookup label. |
| `skip_max_create_reached` | Creation skipped because `--max-create` was reached. |
| `would_link` | Dry-run planned hierarchy issue link. |
| `linked` | Hierarchy issue link was created. |
| `skip_link_existing_state` | Link was already recorded in local state. |
| `skip_link_missing_issue_key` | Link could not be created because parent or child issue key was unavailable. |
| `error` | The row failed. See `message`. |

## HTTP behavior

| Flag | Default | Description |
|---|---:|---|
| `--timeout` | `30` | Per-request HTTP timeout in seconds. |
| `--retries` | `2` | Retry count for transient Jira/AI failures. |
| `--retry-delay` | `2.0` | Base retry delay in seconds. |
| `--jql-label-batch-size` | `50` | Number of lookup labels per Jira search request. |
| `--debug` | off | Prints retry and debug output to stderr. |

## Flat findings dedupe model

Each flat finding is keyed by `rollup_key`.

The script computes:

```text
rollup_external_id = sha256(rollup_key)
rollup_label       = dedupe.label_prefix + first N chars of rollup_external_id
```

Jira lookup uses the generated label, while local state uses the full SHA-256 hash.

Labels are normalized for Jira:

- trimmed
- lowercased
- non-`a-z`, non-`0-9`, non-underscore, and non-hyphen characters become `_`
- repeated underscores collapse to one
- leading/trailing underscores are removed
- empty labels become `unknown`

This allows reruns to skip existing issues and prevents duplicate Jira creation.

## Hierarchy dedupe model

Hierarchy mode uses each plan node’s `external_id` as the local state key.

The Jira lookup label is:

1. `lookup_label` from the hierarchy node, when present; otherwise
2. the node `external_id`

The lookup label is normalized using the same Jira label rules as flat findings.

Hierarchy issue links are cached separately in local state using:

```text
parent_external_id -> child_external_id : issue_link_type
```

This prevents duplicate issue links on reruns.

## Description formats

| Format | Description |
|---|---|
| `wiki` | Sends Jira wiki markup as plain text. Default. |
| `adf` | Sends Atlassian Document Format. Useful for Jira Cloud/API v3 behavior. |
