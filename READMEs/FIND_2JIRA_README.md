# `findings_to_jira.py`

Publish Black Duck vulnerability rollup data into Jira.

The tool supports two publishing modes:

1. **Flat findings mode**: create one Jira issue per unique rollup finding from `findings.csv`.
2. **Hierarchy plan mode**: create Jira Epic/Story/Vulnerability issues from `jira-hierarchy-plan.json`.

Hierarchy plan mode is the recommended workflow when parent/child Jira structure matters.

## Purpose

```text
findings.csv
    ↓
findings_hierarchy_plan.py
    ↓
jira-hierarchy-plan.json
    ↓
findings_to_jira.py
    ↓
Jira issues / links
```

## Inputs

| File | Purpose |
|---|---|
| `findings.csv` | Rollup findings generated by `subp_vuln_rollup.py`. Used in flat findings mode. |
| `jira-hierarchy-plan.json` | Optional hierarchy plan generated by `findings_hierarchy_plan.py`. Used when `--hierarchy-plan` is supplied. |
| `jira-rollup-config.json` | Jira project, issue type, labels, priority/status mapping, dedupe, AI, and hierarchy settings. |
| `jira-rollup-state.json` | Local cache of created/found Jira issues and hierarchy links. |

## Auth

Use environment variables for Jira credentials.

### Basic auth

```bash
export JIRA_USER="user@example.com"
export JIRA_API_TOKEN="..."
```

### Bearer / PAT auth

```bash
export JIRA_PAT="..."
```

Set this in the config file when using `JIRA_PAT`:

```json
{
  "jira": {
    "auth_mode": "bearer"
  }
}
```

`JIRA_URL` can override `jira.url` from the config file:

```bash
export JIRA_URL="https://your-domain.atlassian.net"
```

If Jira URL/auth is incomplete, the tool automatically runs in dry-run mode.

## Common hierarchy dry run

Dry run is the default unless `--apply` is provided.

Installed command:

```bash
blackduck-findings-to-jira \
  --hierarchy-plan jira-hierarchy-plan.json \
  --config src/unofficialblackduck-harness/config/jira-rollup-config.json \
  --state jira-rollup-state.json \
  --results-out jira-hierarchy-publish-results.csv \
  --plan-out jira-hierarchy-publish-plan.json
```

Direct script run:

```bash
python findings_to_jira.py \
  --hierarchy-plan jira-hierarchy-plan.json \
  --config src/unofficialblackduck-harness/config/jira-rollup-config.json \
  --state jira-rollup-state.json \
  --results-out jira-hierarchy-publish-results.csv \
  --plan-out jira-hierarchy-publish-plan.json
```

## Common hierarchy apply run

Creates Jira issues and configured issue links.

```bash
blackduck-findings-to-jira \
  --hierarchy-plan jira-hierarchy-plan.json \
  --config src/unofficialblackduck-harness/config/jira-rollup-config.json \
  --state jira-rollup-state.json \
  --results-out jira-hierarchy-publish-results.csv \
  --plan-out jira-hierarchy-publish-plan.json \
  --apply
```

Hierarchy nodes are processed in dependency order:

1. Epic
2. Story
3. Vulnerability

This allows child nodes to reference parent issue keys when using `jira_parent`, `epic_link_field`, or `issue_link`.

## Flat findings mode

Flat mode creates one Jira issue per unique rollup finding from `findings.csv`.

### Dry run

```bash
blackduck-findings-to-jira \

  --state jira-rollup-state.json \
  --results-out jira-rollup-results.csv \
  --plan-out jira-rollup-plan.json
```

### Apply run

```bash
blackduck-findings-to-jira \
  --findings findings.csv \
```

## Parent modes

| Mode | Description |
|---|---|
| `jira_parent` | Adds Jira `parent.key` to the issue create payload. |
| `issue_link` | Creates the issue first, then links it to its parent using `hierarchy.issue_link_type`. |
| `epic_link_field` | Sets a configured custom Epic Link field. Requires `hierarchy.epic_link_field`. |

Defaults:

- Stories use `jira_parent`.
- Vulnerability issues use `issue_link`.

## Targeting filters

| Flag | Description |
|---|---|
| `--only-parent-project` | Process only findings or hierarchy nodes for this parent project. |
| `--only-parent-version` | Process only findings or hierarchy nodes for this parent version. |
| `--only-subproject` | Process only findings or hierarchy nodes for this subproject. |
| `--only-vulnerability` | Process only findings or hierarchy nodes for this vulnerability ID. |
| `--only-rollup-key` | Flat findings mode only: process one exact raw pipe-delimited `rollup_key`. |
| `--only-rollup-key-hash` | Flat findings mode only: process one finding by full SHA-256 hash of `rollup_key`. |
| `--limit` | Process only the first N findings or hierarchy nodes after filters. |

In hierarchy mode, filters preserve required ancestor nodes so selected child nodes can still attach to their parents.

## Duplicate-stop test

Use the full SHA-256 rollup hash when testing a known Jira duplicate.

```bash
blackduck-findings-to-jira \
  --findings findings.csv \
  --config src/unofficialblackduck-harness/config/jira-rollup-config.json \
  --state jira-rollup-state.json \
  --results-out jira-rollup-results-duplicate-test.csv \
  --plan-out jira-rollup-plan-duplicate-test.json \
  --apply \
  --refresh-existing \
  --only-rollup-key-hash "794762591e2730b5212f6f256c8c38accacbb235c13e577314a56bf7cbf48771" \
  --max-create 1 \
  --debug
```

Expected duplicate-stop summary:

```text
Unique rollup findings:  1
Created:                 0
Skipped existing:        1
Errors:                  0
```

## Jira creation controls

| Flag | Description |
|---|---|
| `--apply` | Actually create Jira issues. Without it, the tool is dry-run only. |
| `--dry-run` | Force dry-run mode even if `--apply` is present. |
| `--refresh-existing` | Query Jira even if the issue exists in local state. Useful for duplicate verification. |
| `--max-create` | Safety cap for the number of Jira issues to create in one run. |
| `--description-format` | `wiki` or `adf`. Use `adf` when required by Jira Cloud/API behavior. |
| `--jql-label-batch-size` | Number of lookup labels per Jira search batch. |

## Output files

| Flag | Default | Description |
|---|---:|---|
| `--results-out` | `jira-rollup-results.csv` | CSV report of each action taken. |
| `--plan-out` | `jira-rollup-plan.json` | JSON dry-run/create plan output for audit/debugging. |
| `--state` | `jira-rollup-state.json` | Persistent local dedupe cache keyed by external ID. |

Common result actions:

| Action | Meaning |
|---|---|
| `would_create` | Dry-run planned issue creation. |
| `created` | Jira issue was created. |
| `created_transition_error` | Jira issue was created, but status transition failed. |
| `skip_existing_state` | Existing issue was found in local state. |
| `skip_existing_jira` | Existing issue was found in Jira by lookup label. |
| `updated_existing_status` | Existing Jira issue was found and moved to configured target status. |
| `existing_status_error` | Existing Jira issue was found, but status transition failed. |
| `skip_max_create_reached` | Creation skipped because `--max-create` was reached. |
| `would_link` | Dry-run planned hierarchy issue link. |
| `linked` | Hierarchy issue link was created. |
| `skip_link_existing_state` | Link was already recorded in local state. |
| `skip_link_missing_issue_key` | Link could not be created because parent or child issue key was unavailable. |
| `error` | The row failed. See `message`. |

## HTTP behavior

| Flag | Default | Description |
|---|---:|---|
| `--timeout` | `30` | Per-request HTTP timeout in seconds. |
| `--retries` | `2` | Retry count for transient Jira/AI failures. |
| `--retry-delay` | `2.0` | Base retry delay in seconds. |
| `--jql-label-batch-size` | `50` | Number of lookup labels per Jira search request. |
| `--debug` | off | Prints retry and debug output to stderr. |

## Optional AI descriptions

AI-generated descriptions are disabled by default.

Enable them in config:

```json
{
  "ai": {
    "enabled": true,
    "base_url": "https://api.openai.com/v1",
    "model": "gpt-4o-mini",
    "api_key_env": "OPENAI_API_KEY"
  }
}
```

Then set the configured environment variable:

```bash
export OPENAI_API_KEY="..."
```

If AI is enabled but the API key is missing, or generation fails, the tool falls back to the built-in Jira wiki description template.

## Flat findings dedupe model

Each flat finding is keyed by `rollup_key`.

The tool computes:

```text
rollup_external_id = sha256(rollup_key)
rollup_label       = dedupe.label_prefix + first N chars of rollup_external_id
```

Jira lookup uses the generated label, while local state uses the full SHA-256 hash.

Labels are normalized for Jira:

- trimmed
- lowercased
- non-`a-z`, non-`0-9`, non-underscore, and non-hyphen characters become `_`
- repeated underscores collapse to one
- leading/trailing underscores are removed
- empty labels become `unknown`

This allows reruns to skip existing issues and prevents duplicate Jira creation.

## Hierarchy dedupe model

Hierarchy mode uses each plan node’s `external_id` as the local state key.

The Jira lookup label is:

1. `lookup_label` from the hierarchy node, when present
2. otherwise the node `external_id`

Hierarchy issue links are cached separately in local state using:

```text
parent_external_id -> child_external_id : issue_link_type
```

This prevents duplicate issue links on reruns.

## Description formats

| Format | Description |
|---|---|
| `wiki` | Sends Jira wiki markup as plain text. Default. |
| `adf` | Sends Atlassian Document Format. Useful for Jira Cloud/API v3 behavior. |

## Notes

- Dry run is the default.
- Add `--apply` only when ready to create or update Jira issues.
- Keep `jira-rollup-state.json` between reruns to avoid duplicate work.
- Use `--refresh-existing` when validating Jira duplicate detection.
