# `findings_hierarchy_plan.py`

Create a normalized Jira hierarchy plan from Black Duck vulnerability rollup findings.

The planner reads `findings.csv`, groups findings into parent/child/vulnerability nodes, and writes JSON/CSV planning files. It does **not** call Jira.

## Purpose

This is the middle stage between raw Black Duck rollup findings and Jira publishing.

```text
parent_projects.csv
    ↓
subp_vuln_rollup.py
    ↓
findings.csv
    ↓
findings_hierarchy_plan.py
    ↓
jira-hierarchy-plan.json
    ↓
findings_to_jira.py
    ↓
Jira issues / links
```

## Inputs

| File | Purpose |
|---|---|
| `findings.csv` | Raw Black Duck rollup findings generated by `subp_vuln_rollup.py`. |

## Outputs

| File | Purpose |
|---|---|
| `jira-hierarchy-plan.json` | Main normalized hierarchy plan consumed by the Jira publisher. |
| `jira-hierarchy-summary.csv` | Human-readable summary of Epic and Story nodes. |
| `jira-hierarchy-nodes.csv` | Flat CSV of every generated node, including vulnerability nodes. |

## Hierarchy model

The planner creates three node types:

| Node type | Logical Jira issue | Grouping |
|---|---|---|
| `epic` | Epic or parent container | One per parent project/version. |
| `story` | Story or child container | One per parent + subproject/version/path. |
| `vulnerability` | Task/Sub-task/link target | One per unique `rollup_key`. |

Logical shape:

```text
Parent Project / Version
└── Epic node
    ├── Story node: Subproject / Version
    │   ├── Vulnerability node
    │   ├── Vulnerability node
    │   └── Vulnerability node
    └── Story node: Subproject / Version
        └── Vulnerability node
```

The actual Jira issue types are configured later in `jira-rollup-config.json` and applied by `findings_to_jira.py`.

## Common run

Installed command:

```bash
blackduck-hierarchy-plan \
  --findings findings.csv \
  --plan-out jira-hierarchy-plan.json \
  --summary-out jira-hierarchy-summary.csv \
  --nodes-out jira-hierarchy-nodes.csv
```

Direct script run:

```bash
python findings_hierarchy_plan.py \
  --findings findings.csv \
  --plan-out jira-hierarchy-plan.json \
  --summary-out jira-hierarchy-summary.csv \
  --nodes-out jira-hierarchy-nodes.csv
```

## Targeting filters

| Flag | Description |
|---|---|
| `--only-parent-project` | Include only findings for this exact parent project name. |
| `--only-parent-version` | Include only findings for this exact parent version. |
| `--only-subproject` | Include only findings for this exact subproject name. |
| `--only-vulnerability` | Include only findings for this exact vulnerability ID. |
| `--limit` | Process only the first N deduped findings after filters. |

Filters are applied before nodes are built.

## Output controls

| Flag | Default | Description |
|---|---:|---|
| `--findings` | `findings.csv` | Input findings CSV. |
| `--plan-out` | `jira-hierarchy-plan.json` | Main hierarchy plan JSON output. |
| `--summary-out` | `jira-hierarchy-summary.csv` | Summary CSV output for Epic/Story nodes. Use an empty string to skip. |
| `--nodes-out` | `jira-hierarchy-nodes.csv` | Flat node CSV output. Use an empty string to skip. |
| `--hash-length` | `24` | Short hash length used in generated lookup labels. |
| `--debug` | off | Prints debug details to stderr. |

## Required findings CSV fields

The input `findings.csv` must include:

```text
parent_project,parent_version,parent_version_href,
subproject_path,subproject,subproject_version,subproject_version_href,
relationship_detection_method,
component,component_version,
vulnerability,score_field,score,severity,
blackduck_url,rollup_key
```

If `rollup_key` is blank, the planner derives one from:

```text
parent_project|parent_version|subproject|subproject_version|component|component_version|vulnerability
```

## Dedupe model

The planner dedupes findings by `rollup_key`.

It then creates deterministic external IDs:

```text
Epic:
  bd_parent_<sha256(parent_project, parent_version, parent_version_href)>

Story:
  bd_child_<sha256(parent_project, parent_version, parent_version_href,
                  subproject, subproject_version, subproject_version_href,
                  subproject_path)>

Vulnerability:
  bd_vuln_<sha256(rollup_key)>
```

Each node also gets a shorter `lookup_label` based on the external ID and `--hash-length`.

The Jira publisher uses these IDs/labels for stable reruns and duplicate prevention.

## Labels

Each generated node includes labels.

Base labels include:

```text
blackduck
subproject_rollup
```

Node-type labels include:

```text
bd_rollup_parent
bd_rollup_child
bd_rollup_vuln
```

Vulnerability nodes also receive a severity label:

```text
bd_sev_critical
bd_sev_high
bd_sev_medium
bd_sev_low
bd_sev_unknown
```

The Jira publisher may add more labels from `jira-rollup-config.json`.

## Node stats

Epic and Story nodes include aggregate stats such as:

```text
finding_count
component_count
vulnerability_count
critical_count
high_count
medium_count
low_count
unknown_count
max_score
min_score
average_score
```

Epic nodes also include:

```text
child_count
```

Vulnerability nodes include per-finding severity counts and `finding_count = 1`.

## Summary CSV

`jira-hierarchy-summary.csv` includes only Epic and Story nodes.

It is intended for quick human review before publishing to Jira.

Common fields include:

```text
node_type,external_id,parent_external_id,lookup_label,summary,
parent_project,parent_version,parent_version_href,
subproject,subproject_version,subproject_version_href,subproject_path,
child_count,finding_count,component_count,vulnerability_count,
critical_count,high_count,medium_count,low_count,unknown_count,max_score
```

## Nodes CSV

`jira-hierarchy-nodes.csv` includes every generated node.

Common fields include:

```text
node_type,external_id,parent_external_id,lookup_label,summary,labels,
parent_project,parent_version,parent_version_href,
subproject,subproject_version,subproject_version_href,subproject_path,
component,component_version,vulnerability,severity,score,
blackduck_url,rollup_key,
finding_count,critical_count,high_count,medium_count,low_count,unknown_count
```

## Jira publisher handoff

The generated `jira-hierarchy-plan.json` is consumed by `findings_to_jira.py`:

```bash
blackduck-findings-to-jira \
  --hierarchy-plan jira-hierarchy-plan.json \
  --config src/unofficialblackduck-harness/config/jira-rollup-config.json \
  --state jira-rollup-state.json \
  --results-out jira-hierarchy-publish-results.csv \
  --plan-out jira-hierarchy-publish-plan.json
```

Dry run is the default. Add `--apply` when ready to create Jira issues/links.

The publisher validates that hierarchy nodes contain:

```text
external_id
node_type
nodes[]
```

Supported node types are:

```text
epic
story
vulnerability
```

## Relationship to Jira config

The planner creates neutral hierarchy nodes.

The Jira publisher later decides:

| Config area | Purpose |
|---|---|
| `hierarchy.epic_issue_type` | Jira issue type for `epic` nodes. |
| `hierarchy.story_issue_type` | Jira issue type for `story` nodes. |
| `hierarchy.vulnerability_issue_type` | Jira issue type for `vulnerability` nodes. |
| `hierarchy.story_parent_mode` | How Story nodes attach to Epic nodes. |
| `hierarchy.vulnerability_parent_mode` | How Vulnerability nodes attach to Story nodes. |
| `hierarchy.issue_link_type` | Jira link type when using `issue_link`. |
| `hierarchy.labels` | Additional labels by node type. |
| `hierarchy.additional_fields` | Additional Jira fields by node type. |

## Notes

- This script does **not** require Jira credentials.
- This script does **not** create Jira issues.
- This script does **not** call Black Duck.
- It only normalizes `findings.csv` into a reusable hierarchy plan.
- The Jira publisher later consumes `jira-hierarchy-plan.json` and creates/links issues in dependency order.
