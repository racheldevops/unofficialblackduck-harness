# `findings_hierarchy_plan.py`

Create a normalized Jira hierarchy plan from Black Duck vulnerability rollup findings.

The script reads `findings.csv`, groups findings into a logical Jira hierarchy, writes JSON/CSV planning files, and does **not** call Jira.

## Purpose

This script is the middle stage between raw vulnerability rollup data and Jira publishing.

```text
findings.csv
    ↓
findings_hierarchy_plan.py
    ↓
jira-hierarchy-plan.json
    ↓
Jira hierarchy publisher
```

## Inputs

| File | Purpose |
|---|---|
| `findings.csv` | Raw Black Duck rollup findings generated by the vulnerability rollup workflow. |

## Outputs

| File | Purpose |
|---|---|
| `jira-hierarchy-plan.json` | Main normalized hierarchy plan for Jira publishing. |
| `jira-hierarchy-summary.csv` | Human-readable summary of Epic and Story nodes. |
| `jira-hierarchy-nodes.csv` | Flat CSV of every generated node, including vulnerability nodes. |

## Hierarchy model

The planner creates three node types:

| Node type | Logical Jira issue | Grouping |
|---|---|---|
| `epic` | Epic | One per parent project/version. |
| `story` | Story | One per parent + subproject/version. |
| `vulnerability` | Task/Sub-task/link target | One per unique `rollup_key`. |

Logical shape:

```text
Parent Project / Version
└── Epic
    ├── Story: Subproject / Version
    │   ├── Vulnerability node
    │   ├── Vulnerability node
    │   └── Vulnerability node
    └── Story: Subproject / Version
        └── Vulnerability node
```

## Common run

```bash
python findings_hierarchy_plan.py --findings findings.csv --plan-out jira-hierarchy-plan.json --summary-out jira-hierarchy-summary.csv --nodes-out jira-hierarchy-nodes.csv
```

## Small parent test

Use this to generate a focused hierarchy plan for one parent project/version.

```bash
python findings_hierarchy_plan.py --findings findings.csv --only-parent-project "cc-goat" --only-parent-version "v2" --plan-out jira-hierarchy-plan-cc-goat.json --summary-out jira-hierarchy-summary-cc-goat.csv --nodes-out jira-hierarchy-nodes-cc-goat.csv
```

## Limited smoke test

Use this to quickly validate output shape without processing the full CSV.

```bash
python findings_hierarchy_plan.py --findings findings.csv --limit 25 --plan-out jira-hierarchy-plan-test.json --summary-out jira-hierarchy-summary-test.csv --nodes-out jira-hierarchy-nodes-test.csv
```

## Targeting filters

| Flag | Description |
|---|---|
| `--only-parent-project` | Include only findings for an exact parent project name. |
| `--only-parent-version` | Include only findings for an exact parent version. |
| `--only-subproject` | Include only findings for an exact subproject name. |
| `--only-vulnerability` | Include only findings for an exact vulnerability ID. |
| `--limit` | Process only the first N deduped findings after filters. |

## Output controls

| Flag | Default | Description |
|---|---:|---|
| `--findings` | `findings.csv` | Input findings CSV. |
| `--plan-out` | `jira-hierarchy-plan.json` | Main hierarchy plan JSON output. |
| `--summary-out` | `jira-hierarchy-summary.csv` | Summary CSV output. Use an empty string to skip. |
| `--nodes-out` | `jira-hierarchy-nodes.csv` | Flat node CSV output. Use an empty string to skip. |
| `--hash-length` | `24` | Short hash length used in generated lookup labels. |
| `--debug` | off | Prints debug details to stderr. |

## Dedupe model

The script dedupes findings by `rollup_key`.

It then creates deterministic external IDs:

```text
Epic:
  bd_parent_<sha256(parent_project, parent_version, parent_version_href)>

Story:
  bd_child_<sha256(parent + subproject/version/path)>

Vulnerability:
  bd_vuln_<sha256(rollup_key)>
```

Each node also gets a shorter `lookup_label` for Jira label-based search.

## Plan JSON shape

The main JSON output contains:

```json
{
  "schema_version": 1,
  "generated_at": "2026-06-26T00:00:00Z",
  "source_findings": "findings.csv",
  "source_counts": {},
  "filters": {},
  "node_counts": {},
  "nodes": []
}
```

Each node contains:

```json
{
  "node_type": "epic",
  "external_id": "bd_parent_...",
  "lookup_label": "bd_parent_...",
  "parent_external_id": "",
  "summary": "...",
  "description": "...",
  "labels": [],
  "context": {},
  "stats": {}
}
```

## Notes

- This script does **not** require Jira credentials.
- This script does **not** create Jira issues.
- It only normalizes `findings.csv` into a reusable hierarchy plan.
- The Jira publisher can later consume `jira-hierarchy-plan.json` and create/link issues in dependency order.
