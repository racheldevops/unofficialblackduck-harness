# `findings_hierarchy_plan.py`

Create a normalized Jira hierarchy plan from Black Duck vulnerability rollup findings.

The planner reads `findings.csv`, groups findings into Jira-ready hierarchy nodes, and writes JSON/CSV planning files. It does **not** call Jira and does **not** call Black Duck.

## Purpose

This is the middle stage between raw Black Duck rollup findings and Jira publishing.

```text
parent_projects.csv
    ↓
subp_vuln_rollup.py
    ↓
findings.csv
    ↓
findings_hierarchy_plan.py
    ↓
jira-hierarchy-plan.json
    ↓
findings_to_jira.py
    ↓
Jira issues / links
```

## Default hierarchy model

The default hierarchy mode is:

```text
vulnerability-project
```

Default Jira shape:

```text
Epic: CVE / vulnerability
└── Task: CVE + directly affected Black Duck project/version
```

Example:

```text
[Black Duck] CVE-2018-1000620
└── CVE-2018-1000620 Project juicy_cam.juiced 1.0.0
```

In this mode:

| Node type | Jira meaning | Grouping |
|---|---|---|
| `epic` | CVE/vulnerability Epic | One per vulnerability ID. |
| `story` | Affected project-version Task | One per vulnerability + affected project + affected version + affected version URL. |
| `vulnerability` | Not generated by default | Count is `0`. |

The planner still uses node type `story` for publisher compatibility, but in `vulnerability-project` mode that node is intended to become a Jira `Task`.

## Affected project/version source

The affected project/version is the directly affected Black Duck project version from each rollup finding:

```text
affected_project = subproject
affected_version = subproject_version
affected_project_version_href = subproject_version_href
```

Parent project/version remains in metadata, descriptions, CSV outputs, and filters for traceability. It does not drive the default Jira hierarchy.

## Legacy hierarchy model

The old project-centered hierarchy remains available with:

```text
project-subproject-vulnerability
```

Legacy shape:

```text
Epic: parent project/version
└── Story: child/subproject version
    └── Vulnerability issue/subtask
```

Use this mode if you want product/project-centered remediation tracking instead of CVE-centered remediation tracking.

## Inputs

| File | Purpose |
|---|---|
| `findings.csv` | Raw Black Duck rollup findings generated by `subp_vuln_rollup.py`. |

## Outputs

| File | Purpose |
|---|---|
| `jira-hierarchy-plan.json` | Main normalized hierarchy plan consumed by the Jira publisher. |
| `jira-hierarchy-summary.csv` | Human-readable summary of Epic and Story/Task nodes. |
| `jira-hierarchy-nodes.csv` | Flat CSV of every generated node. |

## Common default run

Installed command:

```bash
blackduck-hierarchy-plan \
  --findings findings.csv \
  --hierarchy-mode vulnerability-project \
  --plan-out jira-hierarchy-plan.json \
  --summary-out jira-hierarchy-summary.csv \
  --nodes-out jira-hierarchy-nodes.csv
```

`--hierarchy-mode vulnerability-project` is the default, so this is equivalent:

```bash
blackduck-hierarchy-plan \
  --findings findings.csv \
  --plan-out jira-hierarchy-plan.json \
  --summary-out jira-hierarchy-summary.csv \
  --nodes-out jira-hierarchy-nodes.csv
```

Direct script run:

```bash
python src/unofficialblackduck-harness/findings_hierarchy_plan.py \
  --findings findings.csv \
  --hierarchy-mode vulnerability-project \
  --plan-out jira-hierarchy-plan.json \
  --summary-out jira-hierarchy-summary.csv \
  --nodes-out jira-hierarchy-nodes.csv
```

## Focused CVE/project-version test

Generate a focused default hierarchy plan for one parent context, subproject, and CVE:

```bash
blackduck-hierarchy-plan \
  --findings findings.csv \
  --hierarchy-mode vulnerability-project \
  --plan-out jira-hierarchy-plan.json \
  --summary-out jira-hierarchy-summary.csv \
  --nodes-out jira-hierarchy-nodes.csv \
  --only-parent-project "cc-goat" \
  --only-parent-version "v2" \
  --only-subproject "juicy_cam.juiced" \
  --only-vulnerability "CVE-2018-1000620"
```

Expected shape:

```text

Project-version Tasks:   1 or more
Vulnerability nodes:     0
```

## Targeting filters

| Flag | Description |
|---|---|
| `--only-parent-project` | Include only findings for this exact parent project name. |
| `--only-parent-version` | Include only findings for this exact parent version. |
| `--only-subproject` | Include only findings for this exact subproject name. In default mode, this means affected project. |
| `--only-vulnerability` | Include only findings for this exact vulnerability ID. |
| `--limit` | Process only the first N deduped findings after filters. |

Filters are applied before nodes are built.

In default `vulnerability-project` mode:

- `--only-vulnerability` includes the matching CVE Epic and all remaining affected project-version Tasks for that CVE.
- `--only-subproject` includes Tasks where affected project equals the subproject filter and includes the required parent CVE Epics.
- `--only-parent-project` and `--only-parent-version` filter by parent context first, then the remaining findings are grouped into the CVE/project-version hierarchy.

## Output controls

| Flag | Default | Description |
|---|---:|---|
| `--findings` | `findings.csv` | Input findings CSV. |
| `--hierarchy-mode` | `vulnerability-project` | Hierarchy mode. Use `project-subproject-vulnerability` for legacy behavior. |
| `--plan-out` | `jira-hierarchy-plan.json` | Main hierarchy plan JSON output. |
| `--summary-out` | `jira-hierarchy-summary.csv` | Summary CSV output for Epic/Story nodes. Use an empty string to skip. |
| `--nodes-out` | `jira-hierarchy-nodes.csv` | Flat node CSV output. Use an empty string to skip. |
| `--hash-length` | `24` | Short hash length used in generated lookup labels. |
| `--debug` | off | Prints debug details to stderr. |

## Required findings CSV fields

The input `findings.csv` must include:

```text
parent_project,parent_version,parent_version_href,
subproject_path,subproject,subproject_version,subproject_version_href,
relationship_detection_method,
component,component_version,
vulnerability,score_field,score,severity,
blackduck_url,rollup_key
```

If `rollup_key` is blank, the planner derives one from:

```text
parent_project|parent_version|subproject|subproject_version|component|component_version|vulnerability
```

## Default mode grouping and IDs

### CVE Epic

Grouping key:

```text
vulnerability
```

External ID:

```text
bd_cve_<sha256(vulnerability)>
```

Summary:

```text
[Black Duck] CVE-2018-1000620
```

Labels include:

```text
blackduck
subproject_rollup
bd_rollup_cve
bd_cve_<short_hash>
bd_sev_<highest_severity>
```

### Affected project-version Task

Grouping key:

```text
vulnerability|affected_project|affected_version|affected_project_version_href
```

External ID:

```text
bd_cve_project_<sha256(vulnerability, affected_project, affected_version, affected_project_version_href)>
```

Summary:

```text
CVE-2018-1000620 Project juicy_cam.juiced 1.0.0
```

Labels include:

```text
blackduck
subproject_rollup
bd_rollup_project_version
bd_cve_<short_hash>
bd_project_version_<short_hash>
bd_sev_<highest_severity>
```

If the same CVE appears in multiple components for the same affected project/version, the planner creates one Task node and lists all affected components in the Task description.

## Legacy mode IDs

```text

  bd_parent_<sha256(parent_project, parent_version, parent_version_href)>
```

## Jira publisher handoff

The generated `jira-hierarchy-plan.json` is consumed by `findings_to_jira.py`:

```bash
blackduck-findings-to-jira \
  --hierarchy-plan jira-hierarchy-plan.json \
  --config src/unofficialblackduck-harness/config/jira-rollup-config.json \
  --state jira-hierarchy-publish-test-state.json \
  --results-out jira-hierarchy-publish-test-results.csv \
  --plan-out jira-hierarchy-publish-test-plan.json \
  --dry-run \
  --debug
```

Dry run is the default. Add `--apply` when ready to create Jira issues/links.

## Relationship to Jira config

The planner creates neutral hierarchy nodes. The Jira publisher later decides issue types and parent relationships.

Default config for the new hierarchy:

```json
{
  "hierarchy": {
    "epic_issue_type": "Epic",
    "story_issue_type": "Task",
    "vulnerability_issue_type": "Subtask",
    "story_parent_mode": "jira_parent",
    "vulnerability_parent_mode": "jira_parent",
    "issue_link_type": "Relates",
    "epic_link_field": "",
    "labels": {
      "epic": ["bd_rollup_cve"],
      "story": ["bd_rollup_project_version"],
      "vulnerability": ["bd_rollup_vuln"]
    }
  }
}
```

## Notes

- This script does **not** require Jira credentials.
- This script does **not** create Jira issues.
- This script does **not** call Black Duck.
- It only normalizes `findings.csv` into a reusable hierarchy plan.
- The Jira publisher later consumes `jira-hierarchy-plan.json` and creates/links issues in dependency order.
