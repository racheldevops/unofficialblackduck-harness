# `policy_vuln_pull.py`

Pull detailed high-risk Black Duck vulnerability findings for project-version candidates generated by `policy_vuln_find.py`.

## Purpose

```text
policy_candidate_projects.csv
    ↓
policy_vuln_pull.py
    ↓
policy_findings.csv
    ↓
findings_to_datadog.py
```

## Recommended production run

```bash
blackduck-policy-vuln-pull \
  --insecure \
  --candidates policy_candidate_projects.csv \
  --threshold 8.9 \
  --score-operator gt \
  --require-exploit-available \
  --out policy_findings.csv \
  --failures-out policy_pull_failures.csv \
  --api-cache policy_vuln_pull_cache.json \
  --workers 4 \
  --progress-every 10 \
  --cache-save-every 25
```

## Performance controls

| Flag | Default | Description |
|---|---:|---|
| `--workers` | `4` | Concurrent candidate detail pulls. Values greater than `8` are clamped to `8`. Use `1` for sequential behavior. |
| `--progress-every` | `10` | Print progress to stderr every N completed candidates. |
| `--cache-save-every` | `25` | Save API cache and partial findings every N completed candidates. |
| `--partial-out` | `<out>.partial` | Partial findings CSV/JSON output path during pull. |
| `--max-runtime-minutes` | unset | Stop scheduling additional candidates after this runtime and write completed output. |
| `--skip-policy-rules` | off | Avoid following policy-rule links. Cannot be combined with `--policy-name` or `--policy-rule-id`. |
| `--include-policy-rule-details` | off | Follow policy-rule links even when no policy filter is supplied. This can be slower. |

By default, the puller skips policy-rule traversal unless a policy filter is supplied.

## API cache

The persistent API cache is enabled by default:

```bash
--api-cache policy_vuln_pull_cache.json
--api-cache-max-age-hours 20
--api-cache-max-entries 5000
```

For a fully fresh pull:

```bash
--refresh-api-cache
```

The cache is saved periodically according to `--cache-save-every` and again at shutdown.

## Output fields

```text
project,project_version,project_href,project_version_href,
project_group_key,project_group_external_id,
candidate_key,candidate_external_id,
component,component_version,component_origin_id,
vulnerability,severity,score_field,score,
exploit_available,exploitable,reachable,reachability,reachability_source,
policy_name,policy_rule_href,policy_matched,
blackduck_url,bom_component_url,
finding_key,finding_external_id,first_seen_source
```

## Failure handling

A failed candidate does not stop the whole pull. Failures are written to `--failures-out` when supplied.

Failure fields:

```text
project,project_version,project_version_href,candidate_external_id,stage,error
```
