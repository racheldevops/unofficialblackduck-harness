# `subp_vuln_rollup.py`

Roll up Black Duck vulnerabilities from child/subproject project versions into a parent project/version context.

This script reads parent/child project-version relationships, checks vulnerable components on the child versions, filters vulnerabilities by score, and writes normalized rollup findings.

It does **not** call Jira.

## Purpose

```text
parent_projects.csv
    ↓
subp_vuln_rollup.py
    ↓
findings.csv
    ↓
findings_hierarchy_plan.py
    ↓
jira-hierarchy-plan.json
    ↓
findings_to_jira.py
```

## Auth

Use either environment variables or CLI flags.

```bash
export BLACKDUCK_URL="https://blackduck.example.com"
export BLACKDUCK_API_TOKEN="..."
```

### Equivalent flags

```bash

```

## Common run

Installed command:

```bash
blackduck-vuln-rollup \
  --parents-csv parent_projects.csv \
  --threshold 7 \
  --out findings.csv \
  --timeout 30 \
  --retries 1 \
  --page-limit 500 \
  --failures-out failed-rollup-relationships.csv
```

Direct script run:

```bash
python subp_vuln_rollup.py \
  --parents-csv parent_projects.csv \
  --threshold 7 \
  --out findings.csv \
  --timeout 30 \
  --retries 1 \
  --page-limit 500 \
  --failures-out failed-rollup-relationships.csv
```

Use `--insecure` only if needed for lab/on-prem TLS testing.

## Modes

### CSV mode

CSV mode is the recommended workflow.

It uses parent/child relationships from `parent_projects.csv`, usually generated by `find_parent_projects.py`.

```bash
--parents-csv parent_projects.csv
```

Optional parent filters:

```bash
--parent-project "Parent Name"
--parent-version "1.0"
```

Example:

```bash
blackduck-vuln-rollup \
  --parents-csv parent_projects.csv \
  --parent-project "Parent Name" \
  --parent-version "1.0" \
  --threshold 7 \
  --out findings-parent-test.csv
```

### Single parent discovery mode

Single parent mode discovers subprojects from one parent project/version BOM.

```bash
--parent-project "Parent Name"
--parent-version "1.0"
--depth 2
```

Optional BOM-name fallback:

```bash
--resolve-bom-names
```

Example:

```bash
blackduck-vuln-rollup \
  --parent-project "Parent Name" \
  --parent-version "1.0" \
  --depth 2 \
  --resolve-bom-names \
  --threshold 7 \
  --out findings-single-parent.csv
```

## Required CSV input fields

When using `--parents-csv`, the CSV must include:

```text
parent_project,parent_version,child_project,child_version,
parent_version_href,child_version_href
```

The discovery tool also writes additional metadata columns, but the above fields are required for rollup.

## Vulnerability filtering

```bash
--threshold 7
--score-field overallScore
```

| Flag | Default | Description |
|---|---:|---|
| `--threshold` | `7.0` | Minimum numeric vulnerability score included in output. |
| `--score-field` | `overallScore` | Vulnerability score field to evaluate. |

Only vulnerabilities with the selected score field greater than or equal to `--threshold` are written.

## Targeting / retry filters

Use these flags to rerun one child relationship or a smaller subset.

```bash
--only-child-href "https://.../api/projects/.../versions/..."
--only-child-project "Child Project"
--only-child-version "Child Version"
```

| Flag | Description |
|---|---|
| `--only-child-href` | Only check the child relationship with this exact child version href. Best retry selector. |
| `--only-child-project` | Only check child relationships with this child project name. |
| `--only-child-version` | Only check child relationships with this child version name. |

`--only-child-href` is the most precise retry selector.

## Output

```bash
--out findings.csv
--json
--failures-out failed-rollup-relationships.csv
```

| Flag | Description |
|---|---|
| `--out findings.csv` | Writes findings to the specified CSV file. |
| `--out -` | Writes findings to stdout. |
| `--json` | Writes findings as JSON instead of CSV. |
| `--failures-out failed-rollup-relationships.csv` | Writes failed child relationship attempts. |

### Main CSV fields

```text
parent_project,parent_version,parent_version_href,
subproject_path,subproject,subproject_version,subproject_version_href,
relationship_detection_method,
component,component_version,
vulnerability,score_field,score,severity,
blackduck_url,rollup_key
```

## Rollup key

Each finding receives a deterministic `rollup_key`:

```text
parent_project|parent_version|subproject|subproject_version|component|component_version|vulnerability
```

This key is used later by hierarchy planning and Jira publishing to dedupe findings across reruns.

## Failed relationships

If a child relationship cannot be loaded or checked, the script continues processing other relationships.

Use:

```bash
--failures-out failed-rollup-relationships.csv
```

Failure CSV fields include:

```text
parent_project,parent_version,
child_project,child_version,child_version_href,
source,stage,
elapsed_seconds,elapsed_human,
timeout_seconds,retries,attempts_per_request,
error
```

When failures occur, the script also prints suggested individual retry commands to stderr.

## HTTP / TLS

```bash
--timeout 30
--retries 1
--retry-delay 2.0
--page-limit 500
--insecure
```

| Flag | Default | Description |
|---|---:|---|
| `--timeout` | `30` | Per-request timeout in seconds. |
| `--retries` | `1` | Retry count for transient/network failures. |
| `--retry-delay` | `2.0` | Base retry delay in seconds. |
| `--page-limit` | `100` | Black Duck API page size for paginated GETs. |
| `--insecure` | off | Disables TLS certificate validation. Useful for lab/on-prem testing only. |

## API cache

Persistent API cache is enabled by default.

```bash
--api-cache subp_vuln_rollup_cache.json
--api-cache-max-age-hours 20
--api-cache-max-entries 5000
```

Disable persistent cache:

```bash
--no-api-cache
```

Force fresh API reads and rebuild the cache:

```bash
--refresh-api-cache
```

| Flag | Default | Description |
|---|---:|---|
| `--api-cache` | `subp_vuln_rollup_cache.json` | Persistent raw API response cache path. |
| `--no-api-cache` | off | Disables persistent cache. In-run GET cache still applies. |
| `--refresh-api-cache` | off | Ignores existing persistent cache and rebuilds from fresh API responses. |
| `--api-cache-max-age-hours` | `20.0` | Maximum age for persistent API cache entries. Use `-1` to never expire entries. |
| `--api-cache-max-entries` | `5000` | Maximum persistent API cache entries to retain. |

### Recommended daily cron behavior

- Default `--api-cache-max-age-hours 20` avoids using yesterday’s vulnerability data on a daily run.
- If a run fails and is rerun shortly after, the same cache can be reused.
- Use `--refresh-api-cache` for guaranteed fresh nightly output.

## Summary output

The script prints a summary like:

```text
Found 123 rolled-up vulnerabilities with overallScore >= 7.0
```

If failures occurred, it also prints failed relationship details and suggested retry commands.

## Notes

- This script does **not** call Jira.
- This script does **not** create Jira issues.
- It only writes Black Duck vulnerability rollup findings.
- Use `find_parent_projects.py` first to generate `parent_projects.csv`.
- Use `findings_hierarchy_plan.py` next to generate `jira-hierarchy-plan.json`.
